[BUG-C1] Python GUI Runtime Crash - self.colors Reference
  File:   python/gui.py:113
  Issue:  Line 113 uses `self.colors["bg"]` but the class only defines
          `self.THEME`. There is no `self.colors` attribute anywhere in the

          builds the action bar frame.
  Fix:    Change `self.colors["bg"]` to `self.THEME["bg"]`
  Impact: COMPLETE GUI BREAKAGE - application will crash on startup.

[BUG-C2] HTML Syntax Error - Stray Characters
  File:   public/index.html:61
  Issue:  Line 61 contains `</style>e>` - the characters `e>` are stray
          text that will be rendered visibly in the browser (appearing as "e>"
          at the top of the page body).
  Fix:    Change `</style>e>` to `</style>`
  Impact: Visual artifact on the web page; unprofessional appearance.

4. HIGH SEVERITY ISSUES
================================================================================

[HIGH-1] i18n Key Inconsistency - Missing Translation Keys
  Files:  src/i18n/ja-JP.json, src/i18n/ru-RU.json, src/i18n/fr-FR.json
  Issue:  zh-CN.json and en-US.json have 59 keys each, but ja-JP.json,
          ru-RU.json, and fr-FR.json only have 57 keys. The missing keys are:
            - settings.deepScan
            - settings.noiseReduction
          These are used in the settings panel toggle labels. When the UI
          language is Japanese, Russian, or French, the toggle labels will
          fall back to showing the raw key names ("settings.deepScan") instead
          of translated text.
  Fix:    Add the two missing keys to ja-JP.json, ru-RU.json, and fr-FR.json.
  Impact: Broken UI text for 3 of 5 supported languages.

[HIGH-2] Version Inconsistency Across Project
  Files:  package.json, DEVELOPER_GUIDE.md, README.md, ROADMAP.md
  Issue:  package.json declares version "1.5.5", but the documentation and
          feature descriptions extensively reference v1.6.0 features (clipboard
          paste, auto-download, exhaustive testing matrix, memory pooling).
          The ROADMAP.md header says "v1.6.0 - UX & Testing Hardened" with all
          items checked. This creates confusion about what version the code
          actually is.
  Fix:    Either bump package.json to "1.6.0" or update docs to reflect 1.5.5.
  Impact: User/installer confusion; npm package version mismatch.

[HIGH-3] CLI Safety - Unsafe mkdirSync Before Validating Input
  File:   src/cli.js:109
  Issue:  The line `if (!existsSync(outputPath) && !params.input.match(...))`
          creates the output directory based on whether the INPUT matches an
          image extension. This logic is inverted/fragile - it should be
          checking the output path characteristics, not the input path, for
          deciding whether to create a directory.
  Fix:    Refactor to check output path characteristics explicitly, or validate
          input existence before creating output directories.
  Impact: May create unexpected directories if input path has no extension.

[HIGH-4] watermarkEngine.js - getCatalogConfig Called with Wrong Argument
  File:   src/core/watermarkEngine.js:155
  Issue:  After pixel detection succeeds, the code calls:
            `getCatalogConfig(pixelDetect.size)`
          But getCatalogConfig expects (width, height) of the image, not a
          single size number. This will never return a valid catalog match,
          meaning the returned `config` object is always null for pixel-
          detected results, even for official resolutions. The tier badge
          in the UI will never display for pixel-detected images.
  Fix:    Change to: `getCatalogConfig(canvas.width, canvas.height)`
  Impact: Tier badge never shows for pixel-detected images (most cases).

[HIGH-5] Userscript Memory Leak - Blob URLs Not Revoked
  File:   src/userscript/index.js:62
  Issue:  After processing, the code creates a new blob URL:
            `imgElement.src = URL.createObjectURL(processedBlob)`
          But this URL is never revoked. On pages with many Gemini images
          (long conversations), each processed image leaks a blob URL.
  Fix:    Store the URL and revoke it when the element is removed from the DOM
          (via MutationObserver) or before replacing it.
  Impact: Gradual memory leak in long Gemini chat sessions.

5. MEDIUM SEVERITY ISSUES
================================================================================

[MED-1] XSS Potential in AuditLog
  File:   src/app.js (AuditLog object, updateStatus function)
  Issue:  While the AuditLog.log() function properly escapes HTML via
          escapeHtml(), the updateStatus() function at line ~380 and the
          createImageCard() innerHTML assignments use unescaped user-controlled
          data (file names). A malicious filename like
          `<img src=x onerror=alert(1)>.png` could execute JavaScript.
  Fix:    Sanitize item.name through escapeHtml() before inserting into
          innerHTML in createImageCard().
  Impact: Potential XSS if a user processes files with malicious names.

[MED-2] No Input Validation in CLI for --input Value
  File:   src/cli.js:99
  Issue:  The `-i` parameter accepts any string without validating the file
          or directory exists before proceeding. Error messages only appear
          deep in the processing loop.
  Fix:    Add early validation after parsing args: check existsSync(inputPath)
          and provide a clear error message before starting processing.

[MED-3] Detector Module-Level State (Mutable Global)
  File:   src/core/detector.js
  Issue:  The functions `detectWatermark._blurBuffer` and
          `detectWatermark._sharedGradientsI/A` are stored as properties on
          the function object itself. While this serves as a memory pooling
          optimization, it creates hidden mutable state that:
          a) Makes unit testing non-deterministic if tests run in parallel
          b) Cannot be reset between test runs
  Fix:    Consider encapsulating in a Detector class instance, or at minimum
          document this as intentional module-level pooling.

[MED-4] catalog.test.js References Non-Existent Property
  File:   tests/catalog.test.js:10
  Issue:  The test template string uses `entry.name` but GEMINI_SIZE_CATALOG
          entries do not have a `name` property. They only have width, height,
          and tier. This produces "undefined" in test names (though tests
          still pass).
  Fix:    Change `entry.name` to `entry.tier` in the test name template.

[MED-5] blendModes.js - Unused Variable alphaRowOffset
  File:   src/core/blendModes.js (in removeWatermark function)
  Issue:  The variable `alphaRowOffset` is computed at line ~28 but never used
          in the loop body (the correct offset is calculated inline).
  Fix:    Remove the unused `const alphaRowOffset = row * width;` line.

[MED-6] .gitignore Has Duplicate Entries
  File:   .gitignore
  Issue:  Contains duplicate entries:
          - `dist/` appears 3 times (lines 3, 11, 18)
          - `.DS_Store` appears 2 times (lines 5, 14)
          - `.history` appears 2 times (lines 8, 15)
  Fix:    Consolidate to single entries.

[MED-7] Package Manager Inconsistency
  Files:  package.json, package-lock.json, pnpm-lock.yaml
  Issue:  package.json declares `"packageManager": "pnpm@10.11.0"`, but the
          repository also contains a package-lock.json (npm lockfile). Having
          both lockfiles can cause dependency resolution conflicts and team
          confusion about which package manager to use.
  Fix:    Remove package-lock.json and rely solely on pnpm-lock.yaml, or
          vice versa. Add the other to .gitignore.

[MED-8] pnpm-lock.yaml Missing sharp and eslint Dependencies
  File:   pnpm-lock.yaml
  Issue:  The pnpm lockfile does not list `sharp` or `eslint` dependencies
          that are present in package.json. This means `pnpm install` may
          produce a different dependency tree than `npm install`.
  Fix:    Regenerate pnpm-lock.yaml with `pnpm install` to include all deps.

6. LOW SEVERITY ISSUES / CODE QUALITY
================================================================================

[LOW-1]  app.js - Massive Function Scope (~500 lines in single init flow)
          Consider splitting processQueue(), processDirectory(), etc. into
          separate modules (e.g., src/batchProcessor.js).

[LOW-2]  app.js - Magic Numbers
          20 * 1024 * 1024 (20MB file limit) should be a named constant.
          Concurrency formula `Math.min(4, Math.max(1, ...))` repeated twice.

[LOW-3]  cli.js - No --version Flag
          The CLI tool lacks a --version or -v flag despite having version
          info in package.json.

[LOW-4]  cli.js - Race Condition in Pool Pattern
          Lines 141-144: The Set-based pool pattern `pool.delete(p)` inside
          the `.then()` could theoretically miss completions. Using
          `Promise.allSettled` would be cleaner.

[LOW-5]  worker.js - No Error Type Discrimination
          The worker sends only `err.message` back. Adding an error type or
          stack trace would improve debugging in production.

[LOW-6]  watermarkEngine.js - Timeout Magic Number
          Worker timeout (15000ms / 500ms for test) is hardcoded inline.
          Should be a named constant or configurable option.

[LOW-7]  i18n.js - Synchronous localStorage in init()
          The locale detection reads localStorage synchronously during module
          initialization. While this works, it couples the module to the
          browser environment at import time.

[LOW-8]  detector.js - Complex Function (346 lines)
          detectWatermark() is very long with deeply nested loops. Consider
          extracting Phase 1 (anchored), Phase 2 (global search), and
          Phase 3 (ranking) into separate named functions.

[LOW-9]  config.js - Missing isOfficial in Heuristic Fallback
          The heuristic fallback in detectWatermarkConfig() does not set
          `isOfficial: false` explicitly. It relies on the property being
          undefined, which works but is inconsistent with the explicit
          `isOfficial: true` from catalog matches.

[LOW-10] No JSDoc @param Types for Core Functions
          While most functions have JSDoc comments, parameter types are
          informal strings rather than proper TypeScript-style annotations.

[LOW-11] utils.js - Hardcoded Chinese Text
          Line in showLoading(): `'AI 正在精准切除水印边缘...'` is hardcoded
          in Chinese rather than using i18n. This text will always appear in
          Chinese regardless of the selected language.

[LOW-12] tests/detector.test.js - Inconsistent Alpha Map Strategy
          The "Full Catalog Tier Verification" tests create their own alpha
          maps with `createMockAlphaMap(s)` and then modify them with
          `alphaMap[i] = 0.05`, while other tests use the module-level
          `alphaMaps` constant. This inconsistency could mask real issues.

7. ARCHITECTURE & DESIGN OBSERVATIONS
================================================================================

POSITIVE:
  + Clean separation of concerns: core/ (algorithm), app.js (UI), cli.js (CLI)
  + Proper use of Web Workers with fallback resilience
  + Good use of async generators for streaming directory processing
  + Memory pooling in detector (blur buffer, gradient buffers)
  + Comprehensive test suite with parameter matrix coverage
  + Consistent coding style (verified by ESLint + Prettier)
  + Single source of truth for catalog data (catalog.js)
  + Worker timeout mechanism with graceful degradation

AREAS FOR IMPROVEMENT:
  - app.js is a "god module" at ~500 lines handling all UI state, batch
    processing, directory mode, settings, clipboard, etc. This should be
    decomposed.
  - The detector module uses function-level state mutation for memory pooling.
    A class-based approach would be more testable.
  - No TypeScript or JSDoc type enforcement. Consider adding .d.ts files or
    migrating to TypeScript for the core algorithm modules.
  - The userscript directly imports from ../core/ which creates a tight coupling.
    Consider a shared "engine facade" that both the web app and userscript use.
