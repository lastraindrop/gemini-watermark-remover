# v2.7.1 Detection and Restoration Hardening

Date: 2026-07-02

## Scope

This change set executes the P0/P1 work derived from the architecture and code
review: reported missed detections, incorrect overlapping candidates, subtle
post-removal bias, SDK/runtime parity, and release verification.

## Root causes closed

1. A high detector confidence could bypass evidence that trial restoration made
   NCC materially worse. The known 48px fixture therefore also accepted a wrong
   overlapping 96px candidate and changed pixels outside the true watermark.
2. Any catalog-backed candidate suppressed adaptive fallback, even when that
   catalog candidate was weak and a relocated global candidate was stronger.
3. The first multi-pass mutation was committed before regression checks applied.
4. Captured Doubao PNG templates contain a small positive RGB/alpha baseline.
   Treating that baseline as physical alpha caused systematic over-correction.
5. The SDK entrypoint statically imported PNG files, so plain Node ESM import
   failed unless a test-only loader was installed.

## Implemented behavior

- Candidate validation rejects material restoration regression unconditionally.
- Weak catalog candidates no longer block adaptive/global fallback.
- Multi-pass rollback covers the first pass when baseline template evidence is
  authoritative; low-NCC natural texture is not allowed to act as a false veto.
- Doubao's 3/255 correction is asset metadata and follows candidates through
  catalog, adaptive, global, worker, CLI, and browser paths. Exact maps may
  explicitly use zero bias.
- SDK types expose `alphaBias`, candidate asset identity, and the
  `restoration-regression` stop reason.
- Browser binary assets are dynamically imported, preserving build-time data URL
  inlining while allowing plain Node to import the SDK.
- CI runs the standard JavaScript gate and Python bridge on every push and pull
  request.

## Precision evidence

Four source/watermarked Doubao pairs (307x167 TL, 401x173 BR, 221x109 TL, and
276x125 BR) were used to sweep a shared alpha baseline. Their optima clustered
around 3.5-4/255. The production correction is the more conservative 3/255.
Every pair improves and the aggregate MAE improvement is required to exceed 5%
by `tests/doubao_paired_precision.test.js`.

## Remaining browser coverage

The in-app browser Node REPL was unavailable, so the fallback Playwright suite
uses installed Chrome locally and bundled Chromium in CI. Production load,
advanced/manual control activation, real fixture upload and processing, compare
interaction, and mobile overflow are covered. Drag/drop, manual canvas drawing,
and batch ZIP download remain future E2E cases.
